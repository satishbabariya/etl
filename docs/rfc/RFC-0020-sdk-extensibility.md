# RFC 0020: SDK and Extensibility

- **Status:** Draft
- **Author:** TBD
- **Created:** 2026-04-21
- **Supersedes:** None
- **Related:** RFC 0001 (Platform Vision), RFC 0005 (Wasm Runtime), RFC 0006 (Connector Protocol), RFC 0012 (Transformation Layer), RFC 0013 (Pipeline DSL)

## Summary

This RFC specifies how external developers extend the platform: the SDK architecture for authoring connectors and transformations, the supported languages and their toolchains, the connector registry model (packaging, signing, versioning, distribution, discovery), the review and certification process for published artifacts, and the economic model for third-party content (free public, paid private, revenue-sharing marketplace). It defines the developer experience end-to-end, from "I want to write a connector" to "my connector is running in production for paying customers."

Extensibility is an integral part of the wedge. Fivetran's moat is connector breadth; replicating their 500+ connectors ourselves is a multi-year engineering commitment we cannot make alone. The answer is to make connector authoring cheap, safe, and attractive for community contributors, partners, and customers with niche needs. This RFC turns "anyone can write a connector" from aspiration into procedure.

## Motivation

RFC 1 committed to a strategy: ship the top 30 connectors ourselves; rely on the ecosystem to fill the long tail. RFC 5 committed to wasm as the sandboxing mechanism that makes community connectors safe. RFC 6 defined the connector protocol that ensures community connectors interoperate. RFC 12 extended this to transformations.

Each of those RFCs was a prerequisite for extensibility; none of them constitutes an SDK. Without an SDK and the surrounding program:

1. **Nobody authors connectors.** Interfaces alone don't produce developers. A good SDK does.
2. **Authored connectors are low-quality.** Without tooling, testing infrastructure, and review processes, connectors ship bugs.
3. **Nobody can find or install authored connectors.** A registry is not optional.
4. **Nobody makes money authoring connectors.** Without an economic model, the long tail gets authored only for love (rare) or panic (rarer).

We need to make authoring cheap, quality high, distribution easy, and economics sustainable.

## Non-Goals

- This RFC does not specify the exact implementations of SDKs in each language. Those are engineering deliverables downstream of this RFC.
- This RFC does not cover authoring of destination loaders. Per RFC 9, loaders are first-party only. We have no plans to open loader authoring to the community at launch.
- This RFC does not cover visual connector builders or no-code authoring. Those are product features on top of the SDK; not architecture.
- This RFC does not cover the ecosystem partnership model at the business level (revenue shares, marketing collaborations, co-sell). Those are contracts; this RFC specifies the technical substrate.
- This RFC does not cover connector deprecation from the registry beyond its technical mechanics. Deprecation policies are operational decisions.

## Design Principles

**Write once, run on any deployment mode.** A connector authored against the SDK runs in hosted, BYOC, and self-hosted modes identically. The SDK does not care where the worker executing it is.

**Language parity where we can afford it, not perfection.** We launch with first-class support for Rust (highest-performance authoring), TypeScript (broadest reach, large pool of SaaS-API-familiar developers), Go (infrastructure engineers' default), and Python (data engineers' default). Other languages work via their wasm toolchains but without first-class SDK polish.

**Local development is first-class.** A developer writing a connector should be able to run it locally against real sources, with full observability, without touching our infrastructure. Deploying to production is a publishing step, not a development step.

**Registry as infrastructure, not as a second platform.** The registry is part of the catalog and uses the same services (catalog, storage, auth, audit). We do not build a parallel system for it.

**Publication is serious.** Published connectors are trusted by customers' production workloads. The barrier to publication is meaningful: signing, review, testing. We're not a npm for connectors — we're a curated registry.

**Economics respect authors.** Authors who publish useful connectors should be able to earn from them. Not as a charity; as a way to align incentives toward quality and maintenance.

## SDK Architecture

### Layered structure

Every language SDK has three layers:

- **Core bindings**: generated from the WIT definitions (RFC 5, RFC 6). Low-level, one-to-one with the host API. Typically autogenerated and not hand-edited.
- **Idiomatic wrapper**: hand-written thin layer that turns the bindings into language-native constructs. `async fn` in Rust; `async`/`await` in TypeScript; goroutines in Go; `async def` in Python.
- **Developer experience layer**: higher-level helpers for common connector patterns (pagination, rate limiting, OAuth, CDC). Optional; experienced developers skip it.

The layering matters because it lets us ship bindings for any wasm-targeting language quickly (core only) while investing in the developer-experience layer for priority languages.

### Core bindings generation

We maintain the WIT definitions as the canonical interface. For each supported language, a build step regenerates bindings:

- **Rust**: `wit-bindgen` generates Rust types from WIT.
- **TypeScript**: `jco` (JavaScript Component tooling) generates TS types.
- **Go**: `wit-bindgen-go` (from TinyGo project).
- **Python**: `componentize-py` for Python→wasm; bindings via generated Python stubs.

These toolchains are not ours — they're the ecosystem standard. We track them and contribute upstream where we find issues.

### Idiomatic wrappers

The wrapper turns WIT types into language-native feel. Example: Rust's `Result<T, E>` vs. WIT's generated error variants; Python's context managers for resources; TypeScript's Promise-based async.

Wrappers are maintained by us. They are small enough that they can be reviewed and updated for every protocol version.

### Developer experience layer

The most differentiating part. Common connector patterns as reusable helpers:

**Pagination helpers:**

```python
# Python SDK
from platform_sdk.patterns import paginate_cursor

async def read_users(ctx):
    async for page in paginate_cursor(
        first_url="/api/users",
        next_selector=lambda resp: resp.headers.get("Link-Next"),
        page_size=100,
    ):
        for user in page.items:
            ctx.emit_row(user)
```

**OAuth helpers:**

```typescript
// TypeScript SDK
import { OAuthSession } from '@platform/sdk';

const session = await OAuthSession.fromContext(ctx);
const response = await session.get('https://api.example.com/users');
// Token refresh handled transparently.
```

**CDC connector scaffolding:**

```rust
// Rust SDK
use platform_sdk::cdc::{CdcConnector, Position, ChangeEvent};

struct MyCdcConnector { /* ... */ }

#[async_trait]
impl CdcConnector for MyCdcConnector {
    type Position = MyPosition;

    async fn snapshot(&mut self, ctx: &Ctx) -> Result<()> { /* ... */ }
    async fn stream(&mut self, from: Self::Position, ctx: &Ctx) -> Result<()> { /* ... */ }
}
```

**Schema derivation helpers:**

Helpers for building Arrow schemas from native language types (Rust structs, Pydantic models, TypeScript interfaces). This closes the "declare schema + produce matching rows" gap that causes the most errors.

## Supported Languages at Launch

### Rust

**Status**: first-class.

**Why**: highest-performance authoring; first-party connectors are Rust; identical code paths means dogfooding ensures quality.

**Ergonomics**: excellent. `wit-bindgen` Rust support is mature; SDK can use Rust's type system fully.

**Target authors**: engineering teams building custom connectors, infrastructure-savvy partners, our own team for first-party connectors.

### TypeScript

**Status**: first-class.

**Why**: broadest reach; SaaS API developers overwhelmingly know JS/TS; the JavaScript connector ecosystem (Singer taps, Airbyte's Node.js connectors) has a ready audience.

**Ergonomics**: good via `jco`. Async/await maps naturally to the connector protocol.

**Target authors**: frontend-trained engineers, SaaS vendors publishing their own official connectors, community contributors.

### Go

**Status**: first-class.

**Why**: infrastructure engineers' default; strong for database and system connectors; TinyGo's wasm support is mature.

**Ergonomics**: reasonable. TinyGo has limitations (no full stdlib); our SDK documents the supported subset.

**Target authors**: platform engineers, customers writing connectors to their internal systems.

### Python

**Status**: first-class (with caveats).

**Why**: data engineers' default; Airflow, dbt, and pandas ecosystem familiarity. Not including Python loses a massive audience.

**Ergonomics**: trickiest of the four. `componentize-py` is maturing; bundle size is larger than compiled languages; some Python idioms don't translate cleanly to wasm.

**Target authors**: data engineering teams with existing Python codebases, academic / research users.

**Caveat**: performance is lower than Rust/Go/TS for CPU-bound work. We document this; we steer performance-critical connectors to other languages.

### Other languages

**Supported but not first-class** at launch:

- **AssemblyScript**: tiny wasm binaries; good for simple connectors.
- **Zig**: growing wasm support.
- **C/C++**: via wasi-sdk; rarely used for connectors but possible.

These lack the developer-experience layer; authors work closer to the core bindings.

**Not planned**:

- **Java / Kotlin**: JVM-to-wasm (TeaVM) exists but is immature for production use.
- **Ruby**: no meaningful wasm target.
- **PHP**: no meaningful wasm target.

## Connector Authoring Workflow

End-to-end: a developer goes from idea to published connector.

### 1. Bootstrap

```bash
platform-cli connector create --language rust my-source-connector
cd my-source-connector
```

Creates a skeleton project with:
- `Cargo.toml` (or equivalent for other languages).
- `src/lib.rs` (or equivalent) with connector-shaped boilerplate.
- `manifest.yaml` with metadata.
- `tests/` with example fixtures.
- `README.md` with scaffolding.

### 2. Develop

Author implements `describe`, `validate-config`, `discover`, `read`. The SDK provides:

- Type hints / autocomplete for every SDK function.
- Hot-reload during local testing.
- Real-time log streaming from local wasm runtime.
- Local catalog (file-backed) for state persistence during development.

### 3. Test locally

```bash
platform-cli connector test
```

Runs the connector's test suite (per RFC 6 publication gates) against local fixtures. Uses the same wasm runtime as production (same tool as hosts the connector in production) so "works locally, breaks in production" is rare.

Developers can also run against real sources:

```bash
platform-cli connector dev \
  --config my-config.yaml \
  --stream users \
  --limit 1000
```

Extracts up to 1000 rows from a real source and displays them. No production impact; no need for platform account.

### 4. Build

```bash
platform-cli connector build
```

Compiles to wasm, runs AOT compilation, produces `.cwasm` artifact. Embeds manifest and signature. Artifact is ready for publication.

### 5. Publish

Requires platform account.

```bash
platform-cli connector publish --registry public
```

Uploads the artifact to the registry. Triggers publication workflow:

- Signature verification.
- Automated tests run on our infrastructure against known-good fixtures.
- Manifest validation (schema compliance, naming conventions, required fields).
- Static analysis (pattern-matching against known-bad code patterns; simple supply-chain checks).
- Registry entry created with status `pending_review` (for public connectors) or `published` (for private).

### 6. Review (public connectors only)

Our team reviews public connectors:

- Code review of the source (required: public connectors ship source).
- Functionality verification.
- Security review (for connectors with elevated capabilities or against sensitive sources).
- Documentation review.

Review target: 10 business days for standard connectors, longer for complex or security-sensitive ones. Rejections come with specific feedback; authors iterate.

Approved connectors transition to `published` and appear in the public registry.

### 7. Maintain

Post-publication:

- Authors get metrics (usage counts, error rates across customers — aggregated only, no per-customer data).
- Bug reports surface through an issue tracker associated with the connector.
- Version updates follow the same publication flow.
- Security advisories from us propagate to authors when relevant.

## Connector Registry

### Registry as catalog extension

The registry is not a separate service — it uses the same catalog (RFC 10), object storage (for artifacts), and auth (RFC 2) as the rest of the platform. Registry-specific concerns (publishing workflow, review queue, search) are Registry Service logic on top of shared infrastructure.

### Entity model

Each published artifact is a `ConnectorVersion` (or `TransformationVersion`) entity per RFC 10. Extensions for registry:

```
ConnectorVersion {
  ... (from RFC 10)
  visibility: enum { private, tenant_shared, organization_shared, public },
  owner_tenant_id: UUID,
  artifact_ref: object-storage-URI,
  signature: bytes,
  review_status: enum { pending, approved, rejected, deprecated },
  review_feedback: Option<string>,
  statistics: RegistryStatistics,
  economic_terms: Option<EconomicTerms>,
}
```

### Visibility levels

**Private**: accessible only to the authoring tenant. Use case: customer-internal connectors they don't want to share.

**Tenant shared**: accessible to a defined set of tenants. Use case: an organization with multiple subsidiaries on our platform.

**Organization shared**: a meta-tenant grouping (e.g., "Acme Corp" parent with multiple child tenants sharing internal connectors).

**Public**: accessible to all tenants; searchable in the public registry.

The default at publication is the most restrictive applicable; escalating to public is an explicit choice.

### Registry API

```
GET /registry/connectors                 # list / search
GET /registry/connectors/{publisher}/{name}         # detail
GET /registry/connectors/{publisher}/{name}/versions  # version list
GET /registry/connectors/{publisher}/{name}/{version} # specific version

# Write operations (auth'd):
POST /registry/connectors                # publish
PATCH /registry/connectors/{publisher}/{name}/{version}/deprecate
DELETE /registry/connectors/{publisher}/{name}/{version} # admin-only
```

Analogous endpoints for transformations.

### Discovery

Customers find connectors via:

- **Browse by category** (database, SaaS, file, custom): UI.
- **Search** by name / publisher / capability (CDC, real-time, etc.).
- **Recommendations** based on other connectors the tenant uses.
- **Integration directory** — curated pages for specific source categories.

Search is backed by a standard full-text search; at our scale it's Postgres FTS. At large scale, a dedicated search service.

## Publishing Quality Gates

A connector cannot publish without passing:

### Automated gates

- **Schema compliance**: manifest conforms to registry schema.
- **Naming conventions**: publisher/name format, no reserved prefixes.
- **Signature present** and valid.
- **Test suite passes**: every test category from RFC 6.
- **Resource budget declared**: the connector declares its CPU/memory needs and they are within platform caps.
- **No known-bad patterns**: static analysis against a ruleset (e.g., disallowed crate dependencies, patterns suggesting attempted sandbox escape).

These run on our infrastructure in a sandbox; they gate every publication attempt.

### Manual gates (public registry)

- **Source available**: public connectors are required to publish their source alongside the compiled artifact. Not open-source license-required; just readable-by-reviewers.
- **Documentation complete**: README, setup guide, supported modes, any source-specific caveats.
- **Security review** (for connectors targeting sensitive source types or requesting elevated capabilities).

Manual gates are a quality bar. Private / tenant-shared connectors are exempt from manual gates (customer's own risk with their own code).

## Versioning and Updates

### Semver discipline

Every connector follows semver. Breaking changes require major bumps:

- Adding a capability: non-breaking.
- Adding optional config fields: non-breaking.
- Adding required config fields, removing fields, changing field semantics, incompatible schema changes: breaking.

Publication enforces this via manifest-diff analysis: changes between versions are classified automatically and the publisher must confirm the classification or accept our reclassification.

### Deprecation

Authors can mark versions as deprecated:

- Existing pipelines continue working on deprecated versions.
- New pipelines pinning the deprecated version receive warnings.
- A deprecation window (default 12 months) applies before version removal.

Abandoned connectors (no updates for 24 months, no author response to issues): we can take maintainership or mark as archived. Explicit policy.

### Upgrade path for customers

Pipelines pin to connector versions (RFC 4, RFC 10). Customer-driven upgrades:

- UI surfaces upgrade prompts for minor versions.
- Major versions require explicit action.
- Upgrade preview: run a new version in preview mode before committing.

## Economic Model

This is where extensibility either succeeds or becomes a charity.

### Free public connectors

Most connectors launch free:

- Our own first-party connectors.
- Community-contributed connectors.
- SaaS vendors publishing official connectors to their services.

"Free" means: no per-use charge from the connector author. Customers still pay us for platform usage (compute, storage, etc., per RFC 17) when they run the connector.

### Paid connectors (future)

For authors who invest meaningfully and want sustainable revenue:

- Author sets a subscription price (monthly per tenant) or per-usage price.
- Customers authorize payment at pipeline configuration time.
- Our platform handles billing; author gets revenue share (typical 70/30 split, negotiable).
- Authors are responsible for maintenance; customers expect support.

Paid connectors are a later feature. Launch is free-only while we build the billing plumbing and establish the author community.

### Enterprise partnerships

Strategic connectors often come from partnerships (e.g., a CRM vendor publishing their official connector):

- Bespoke contracts with SaaS vendors.
- Potentially non-standard revenue-sharing.
- Joint go-to-market.

These are business arrangements; the technical substrate is the same registry.

### Ecosystem fund (possible)

To seed connector breadth at launch, we may offer grants / bounties for priority connectors:

- Identified by customer demand ("we need an X connector but nobody has built it").
- Bounty amount based on estimated effort.
- Authors compete or are invited.

This is a growth-tactic, not a commitment. Launch with ours plus community; accelerate with bounties if needed.

## Support Model for Authored Connectors

### Who supports what

**First-party connectors** (published by us): we support them fully. Standard customer support channels.

**Verified partner connectors** (SaaS vendors publishing their own): partner provides first-line support; we escalate platform-level issues.

**Community connectors**: author provides support per their own preference. We triage issues that turn out to be platform bugs. We do not commit to maintaining abandoned community connectors.

### Issue tracking per connector

Each registry entry has an issue tracker. Authors, users, and our staff can participate. This keeps "my Stripe connector doesn't work" conversations with the connector author, not with our overloaded general support.

### Liability

Legal boilerplate: community connectors are used at customer's own risk; we disclaim liability. Partner and first-party connectors carry our standard SLA commitments.

## Transformation Authoring (Summary)

Most of the above applies equally to transformations (RFC 12). Differences:

- Transformations are generally smaller (a package of declarative ops + optional UDFs).
- Economics are similar but transactions are typically free; paid transformations rarer than paid connectors.
- Testing is simpler (deterministic, pure functions).
- Review standards are lower (less attack surface; transformations can't access network, secrets, or state).

Transformations share the registry infrastructure.

## Integration Partners (Non-Connector)

Beyond connectors, the platform has adjacent integration surfaces worth naming:

### Terraform provider

For customers who manage pipelines via Terraform rather than our CLI/DSL directly. We publish a `platform` provider to the Terraform registry. Resources map to our DSL resources (RFC 13).

### Kubernetes operator

A Kubernetes custom-resource-definition (CRD) + operator for customers who prefer `kubectl` over `platform-cli`. CRDs mirror RFC 13 resources. Same underlying API.

### dbt integration

We cooperate with dbt rather than compete. A dbt-platform adapter lets dbt models reference data loaded by our pipelines as upstream sources, and lets our pipelines trigger downstream dbt runs.

### CI integration examples

GitHub Actions, GitLab CI, CircleCI templates for common workflows:

- Validate YAML on PR.
- Preview diff on PR.
- Apply on merge to main.

These are example templates, not software we operate.

## Alternatives Considered

**Build every connector ourselves (no community authoring).** Simplest. Rejected in RFC 1 and reaffirmed: connector breadth is Fivetran's moat; we need ecosystem leverage.

**Accept connectors written in any language with no SDK investment.** Lower cost for us. Rejected: quality and ergonomics would suffer; community contributors bounce off bad tooling. SDK investment is leverage.

**No review process for public connectors (open npm-style).** More open; attracts more contributors. Rejected: we can't have customers' production workloads depend on unreviewed code. A curated registry is the correct quality bar.

**Make all connectors open-source.** Goodwill; community multiplier. Rejected as requirement: paid private connectors are a legitimate business. Source-available (readable-by-reviewers) is our middle ground for public connectors.

**Force loaders authoring on the community.** Rejected in RFC 9 and reaffirmed here: loader correctness is too load-bearing for community authoring. Loaders are first-party. Adding a destination is an engineering project for us, not a community contribution.

**Host Python via Pyodide for full-fat Python.** Would make Python more capable but much heavier. Rejected: size and startup cost don't pay back. We support componentize-py which is slimmer and sufficient for the connector domain.

## Open Questions

1. **Exact revenue-share percentages.** 70/30 is the likely default for paid connectors; enterprise-tier partners may negotiate. Business decision.
2. **Java/Kotlin support timeline.** The JVM-to-wasm toolchain (TeaVM, CheerpJ) may mature in 2-3 years. Monitor; adopt when viable.
3. **AI-assisted connector authoring.** "Paste API docs; get a connector scaffold" — promising direction; needs careful framing to avoid quality issues. Research in year two.
4. **Private registries for customers.** Large enterprises want their own internal registry. Feasible as tenant-scoped registry views; not a distinct product at launch.
5. **Connector portability across ETL platforms.** Could our SDK become a cross-platform standard (Airbyte-compatible, Singer-compatible)? Ambitious; would require consortium work. Monitor opportunities.
6. **Signed-by-vendor verified partner tier.** A tier higher than "community" but not "platform-published": verified by the original SaaS vendor. Implementable via vendor-specific signing keys registered with us.
